# 实施文档设计说明

> 原子化文档体系设计思路

## 🎯 设计原则

### 1. 原子化
- **一文一事**: 每个文档只聚焦一个独立任务
- **最小单元**: 任务可在 1-6 小时内完成
- **可验收**: 每个任务都有明确的验收标准

### 2. 精简化
- **行数控制**: 200-300 行/篇
- **去除冗余**: 只保留必要的执行步骤
- **快速定位**: 通过编号和标题快速找到目标

### 3. 可执行
- **复制即用**: 代码块可直接复制执行
- **步骤清晰**: 按顺序执行不出错
- **及时验证**: 每步都有验证方法

## 📐 文档结构

### 标准模板

```markdown
# XX - 任务标题

> **阶段**: PhaseX | **时间**: Xh | **前置**: XX

## 🎯 任务目标
[一句话说明任务目的]

## 📋 执行步骤
[3-8 个步骤，每步都有代码示例]

### 1. 步骤一
[说明 + 代码]

### 2. 步骤二
[说明 + 代码]

## ✅ 验收标准
[3-5 个检查点]

### 检查 1: [检查项]
[验证命令 + 预期结果]

## 📝 检查清单
- [ ] 检查项 1
- [ ] 检查项 2

## 🚨 常见问题
[2-3 个常见问题及解决方案]

## 🔗 下一步
[链接到下一个文档]
```

### 元信息说明

- **阶段**: 所属阶段编号
- **时间**: 预计完成时间
- **前置**: 前置依赖文档编号

## 🗂️ 文档组织

### 目录结构
```
implementation-docs/
├── README.md              # 总览和快速开始
├── milestones.md          # 里程碑验收标准
├── document-list.md       # 完整文档列表
├── document-design.md     # 本文件
│
├── phase-1-infrastructure/  # 阶段一：7 个文档
├── phase-2-core-features/   # 阶段二：8 个文档
├── phase-3-main-app/        # 阶段三：9 个文档
├── phase-4-sub-apps/        # 阶段四：8 个文档
├── phase-5-deployment/      # 阶段五：6 个文档
└── phase-6-tooling/         # 阶段六：4 个文档
```

### 命名规范
- 格式: `编号-任务名称.md`
- 编号: 两位数字（01-42）
- 任务名称: 小写英文，用连字符分隔
- 示例: `01-monorepo-init.md`

## 📊 任务拆分策略

### 拆分原则

**1. 按功能模块拆分**
- 坏例子: ❌ `core-features.md`（包含 EPS + CRUD + 插件）
- 好例子: ✅ `vite-plugin-eps.md` + `crud-composable.md` + `plugin-manager.md`

**2. 按实现层次拆分**
- 坏例子: ❌ `crud-system.md`（组件 + 逻辑混在一起）
- 好例子: ✅ `crud-composable.md` + `crud-table-component.md` + `crud-form-component.md`

**3. 按独立性拆分**
- 每个文档产出应该是独立可验证的
- 后续文档可以依赖前面文档的产出
- 避免循环依赖

### 拆分示例

**原任务**: 主应用开发（太大）

**拆分后**:
```
16. 主应用初始化         -> Vue + Vite 环境
17. qiankun 基础配置     -> qiankun 集成
18. 微应用加载器         -> 加载逻辑
19. 全局状态管理         -> Pinia store
20. 布局-头部组件        -> Header 组件
21. 布局-侧边栏组件      -> Sidebar 组件
22. 登录认证            -> Login 页面
23. 路由守卫            -> 权限拦截
24. 系统管理-用户模块    -> 第一个 CRUD
```

## ⏱️ 时间估算

### 估算标准

| 任务类型 | 预计时间 |
|---------|----------|
| 配置类（如 TypeScript） | 1-2h |
| 基础开发（如工具函数） | 2-3h |
| 组件开发（如 CRUD 组件） | 3-4h |
| 系统集成（如 qiankun） | 3-4h |
| 业务模块（如订单管理） | 4-6h |
| 工具开发（如 CLI） | 4-5h |

### 缓冲时间
- 新手: 在预估时间基础上 ×2
- 中级: 在预估时间基础上 ×1.5
- 专家: 按预估时间执行

## 🔄 文档迭代

### 更新策略
1. **初版**: 覆盖所有步骤和验收点
2. **迭代**: 根据实际执行反馈优化
3. **完善**: 补充常见问题和最佳实践

### 反馈机制
- 执行过程中记录问题
- 完成后更新文档
- 定期 review 和优化
